# PRD 템플릿 (8섹션 고정)

이 파일은 PRD 본문의 고정 구조와 항목별 작성 가이드다. 아래 골격을 그대로 채워
출력한다. `<…>`는 채울 자리, `[확인 필요: …]`는 기획서에 근거가 없을 때 그대로 남길
마커다.

---

## 머리말 (메타 표)

문서 맨 위에 한 줄 표로 메타 정보를 둔다.

| 항목 | 내용 |
| --- | --- |
| 화면 ID | `<화면 ID>` |
| 출처 기획서 | `<Figma 파일/노드 또는 문서명>` |
| 작성 기준 | `<읽은 소스 요약: 명세 텍스트 + 목업 N종 등>` |
| 버전 | Draft 0.1 |

머리말 아래에 작성 원칙 한 줄을 적는다:
> 작성 원칙: 기획서에 명시되지 않은 사항은 임의로 채우지 않고 본문 내 `[확인 필요: …]`로
> 표시했으며, 문서 끝 **확인이 필요한 항목**에 모았습니다.

---

## 1. 개요 / 배경 및 문제 정의

- **문제**: 이 기능이 해결하려는 사용자/운영 문제. 기획서에서 근거 가능한 범위로만.
- **목적**: 사용자가 이 화면에서 무엇을 할 수 있게 되는가.
- **대상 화면 범위**: 이 PRD가 다루는 화면 목록.

배경이 기획서에 없으면 화면 명세에서 직접 도출 가능한 만큼만 적고, 비즈니스 배경은
`[확인 필요]`로 둔다.

## 2. 목표 및 성공 지표 (KPI)

측정 가능한 지표를 표로. 기획서에 KPI가 없으면 **제안 지표(초안)**임을 명시하고,
지표 확정을 `[확인 필요]`로 올린다.

| 구분 | 지표 | 측정 방식 |
| --- | --- | --- |
| 사용성 | `<예: 첫 렌더 ≤ 1.5초>` | `<측정 방법>` |
| 참여 | `<예: 열람률>` | `<산식>` |

## 3. 사용자 스토리

`US-n` 형식. "~로서, ~하고 싶다. ~하기 위해서." 구조.

- **US-1** `<역할>`로서, `<목표>`하고 싶다. `<이유>`.

## 4. 기능 요구사항

`FR-n` 형식. 각 항목에 설명 → 우선순위 → **인수 조건(Given-When-Then)** 순.
정상 흐름 + 경계/예외 케이스를 모두 인수 조건에 포함한다.

### FR-1. `<기능명>` — `P0|P1|P2`

`<기능 설명: 구성 요소, 규칙(정렬/표시 등)>`

**인수 조건**
- **Given** `<전제>` **When** `<행동>` **Then** `<기대 결과>`
- **Given** `<경계/예외 전제>` **When** `<행동>` **Then** `<기대 결과>`

`[확인 필요: <이 기능 구현에 필요하나 명세에 없는 것 — API 스펙, 저장 위치 등>]`

(기능 수만큼 FR-2, FR-3 … 반복. 네비게이션·진입점도 별도 FR로.)

## 5. 비기능 요구사항

하위 항목으로 나눠 작성:
- **성능**: 렌더/전환 시간(측정 가능), 스크롤 성능 등. 기준 조건 미정 시 `[확인 필요]`.
- **보안 / 권한**: 인증 필요 여부, 비로그인 접근 가능 여부.
- **접근성**: 동적 폰트 스케일, 색상 외 보조 구분(색각 이상 대응) 등.
- **데이터 / 표시 규칙**: 날짜 포맷, 타임존, 정렬 tie-break 등.

## 6. 범위 외 (Out of Scope)

이 PRD가 다루지 **않는** 것. 기획서에 언급됐지만 본 화면과 무관해 보이는 항목은
여기에 빼고, 동시에 확인 항목(Q)으로도 올려 교차 참조한다.
